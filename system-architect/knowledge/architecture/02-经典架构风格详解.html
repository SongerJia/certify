02 — 经典架构风格详解

对应计划：第 1 周 周二 ~ 周三 | 难度：中等 | 考试权重：选择题 5~8 分 + 案例常客

一、学习目标

  • [ ] 画出分层架构的经典 3 层/4 层结构图
  • [ ] 理解事件驱动架构的发布-订阅机制和适用场景
  • [ ] 区分微内核 vs 宏内核，理解插件化思想
  • [ ] 掌握管道-过滤器的批量/增量处理模式
  • [ ] 理解 SOA 的核心组件（ESB、UDDI、WSDL）和作用

二、分层架构 (Layered Architecture)

2.1 核心思想

将系统按抽象层次组织成 N 层，每层只依赖相邻下层，上层调用下层，下层不依赖上层。

2.2 经典三层结构

┌──────────────────────────────┐
│         表示层 (UI)           │  ← 用户交互
├──────────────────────────────┤
│        业务逻辑层 (BLL)        │  ← 核心业务规则
├──────────────────────────────┤
│        数据访问层 (DAL)        │  ← 数据持久化
├──────────────────────────────┤
│          数据库               │  ← 存储
└──────────────────────────────┘

2.3 四层扩展（DDD 风格）

层次职责关键组件
用户接口层处理用户交互，协议转换Controller, DTO
应用层协调领域对象，事务管理Application Service
领域层核心业务逻辑Entity, Value Object, Domain Service
基础设施层技术支撑Repository 实现, 消息队列, 缓存

2.4 分层架构的优缺点

优点缺点
结构清晰，易于理解性能开销（层层穿透）
层间低耦合，支持替换级联修改风险（底层变更影响所有上层）
有利于标准化和分工严格分层可能导致冗余 DTO 转换
可测试性好（逐层 Mock）不适合高实时性系统

2.5 宽松分层 vs 严格分层

类型规则适用场景
严格分层层 N 只能调用层 N-1大型、多人协作项目
宽松分层层 N 可以调用任意下层追求性能、简单项目

考试要点：案例中常考"严格分层的优缺点"以及"何时应打破分层"。

三、事件驱动架构 (Event-Driven Architecture)

3.1 核心思想

组件之间不直接调用，而是通过事件进行通信。事件生产者发布事件到事件通道，事件消费者订阅并异步处理。

3.2 四种拓扑结构

拓扑特点适用场景
中介者拓扑通过中央事件中介协调复杂事件流编排
代理拓扑点对点事件传输，无中心简单事件链
流处理拓扑事件作为数据流顺序处理实时数据处理
CQRS/事件溯源读写分离 + 事件存储高并发查询系统

3.3 发布-订阅模型

Producer → EventChannel → Consumer1
           (Topic/Queue) → Consumer2
                         → Consumer3
概念说明
Event发生的某件事的状态变化
Publisher产生事件的组件
Subscriber响应事件的组件
Event Channel事件传输媒介

3.4 优缺点

优点缺点
极高松耦合调试困难（事件链路不透明）
高扩展性（加消费者即可）事件顺序一致性难保证
异步提升吞吐量错误处理和恢复复杂
适合异构系统集成可能存在事件风暴

真题关联：2025.05 论文题目之一 "事件驱动架构设计"

3.5 考试常见陷阱

  • ❌ "事件驱动架构一定比分层架构好" — 错误，各有适用场景
  • ❌ "事件驱动就是异步" — 不完全是，关键是"事件"解耦
  • ✅ "EDA 适合需要松耦合、异步处理、高扩展性的系统"

四、微内核架构 (Microkernel Architecture)

4.1 核心思想

核心系统只包含最基础的功能，扩展功能通过插件形式外挂。

4.2 结构

┌─────────────────────────────────────┐
│         插件层 (Plugins)             │
│  ┌──────┐ ┌──────┐ ┌──────┐        │
│  │Plugin│ │Plugin│ │Plugin│ ...     │
│  │  A   │ │  B   │ │  C   │        │
│  └──┬───┘ └──┬───┘ └──┬───┘        │
├─────┼────────┼────────┼─────────────┤
│     │  核心系统 (Core System)        │
│     │  · 插件管理 · 进程调度         │
│     │  · 最小功能集 · 通信机制       │
└─────┴───────────────────────────────┘

4.3 现实例子

系统核心插件
Eclipse IDE运行时框架各语言支持、插件
Chrome 浏览器渲染引擎扩展程序
OSGi 框架模块化运行时Bundle
操作系统内核进程调度/内存管理驱动、文件系统

4.4 微内核 vs 宏内核

维度微内核宏内核
内核大小极小（最小化）庞大（包含驱动等）
扩展性插件式，高需重新编译内核
性能较慢（IPC 开销）快（函数调用）
安全隔离好（故障隔离）差（驱动崩溃影响全局）
代表MINIX、QNX、华为鸿蒙 LiteOSLinux、Windows

4.5 适用场景

  • 产品需要支持多版本/多配置
  • 第三方扩展开发平台
  • 需要热插拔功能
  • 对扩展性要求高，对性能要求适中

五、管道-过滤器架构 (Pipe-Filter)

5.1 核心思想

数据从一个过滤器流向另一个过滤器，每个过滤器独立处理并传递结果。

5.2 经典结构

Input → [Filter1] → Pipe → [Filter2] → Pipe → [Filter3] → Output
         大写转换              去空格                加密

5.3 关键概念

概念说明类比
Filter处理数据的独立单元Unix 命令（grep/sort/awk）
Pipe连接两个过滤器的通道Unix 管道 `
数据流沿管道流动的数据水流

5.4 两种处理模式

模式工作方式例子
批处理等所有输入到达再处理编译器（源文件→词法分析→语法分析→代码生成）
增量处理输入到达即处理视频流实时转码

5.5 优缺点

| 优点 | 缺点 | |:---|:---|:---| | 过滤器高内聚、可复用 | 不适合交互式系统 | | 易于并行化 | 错误处理困难 | | 支持增量演化和替换 | 数据格式转换开销 | | 组合灵活 | 批处理模式延迟高 |

2025.05 真题：案例分析考了"解释器风格"（虚拟机风格的一种），管道-过滤器作为对比选项出现。

六、SOA 架构 (Service-Oriented Architecture)

6.1 核心思想

将业务功能封装为服务，通过标准化的服务契约（接口）进行集成，实现跨系统、跨平台的互操作性。

6.2 SOA 三角色模型

 ┌──────────────────────────────────┐
 │      服务注册中心 (UDDI)          │
 │  ┌────┐  ┌────┐  ┌────┐         │
 │  │支付│  │订单│  │用户│  ...     │
 │  └────┘  └────┘  └────┘         │
 └───┬──────────────┬──────────────┘
     │   ②查找     │ ①发布
     ▼              ▼
 ┌─────────┐   ┌─────────┐
 │服务消费者│──→│服务提供者│
 └─────────┘ ③└─────────┘
            绑定/调用
角色职责对应技术
服务提供者实现并发布服务Web Service 实现
服务消费者发现并调用服务调用方应用
服务注册中心存储服务描述和地址UDDI
企业服务总线 ESB消息路由、协议转换、数据转换Mule, Apache Camel

6.3 SOA 核心原则

原则说明
标准化服务契约服务通过 WSDL 等标准描述
服务松耦合消费者不依赖提供者实现
服务抽象隐藏内部逻辑，只暴露契约
服务可复用面向企业级复用
服务自治服务独立部署、独立演化
服务无状态尽量不保持会话状态
服务可发现通过注册中心动态发现
服务可组合原子服务可组合为复合服务

6.4 SOA vs 微服务

维度SOA微服务
服务粒度较粗（业务域）较细（单一职责）
通信方式ESB 集中式轻量协议（HTTP/gRPC）
数据管理共享数据库常见每个服务自有数据库
治理集中治理去中心化治理
耦合度较松（ESB 成为耦合点）更松（独立部署）

真题关联：2025.05 论文题目之一涉及 SOA 架构

七、5 种风格快速对比

维度分层事件驱动微内核管道-过滤器SOA
组织方式层次事件通道核心+插件管道链服务
耦合度中低低低中
扩展方式加层/改层加消费者加插件加过滤器加服务
典型应用Web 后端IoT/消息IDE/OSETL/编译器企业集成
性能中中中偏低中偏高中
测试难度低高中低高

八、章节练习（25 题）

单选题（15 题）

1. 关于分层架构，下列描述错误的是？

  • A. 每层只依赖其直接下层
  • B. 严格分层比宽松分层更灵活
  • C. 有利于关注点分离
  • D. 上层变化不影响下层
答案B — 严格分层更死板（必须逐层通过），宽松分层更灵活（可跨层调用）

2. 以下哪个不是事件驱动架构的适用场景？

  • A. IoT 设备数据采集
  • B. 电商订单状态流转
  • C. 银行核心记账系统（强一致性事务）
  • D. 日志聚合分析
答案C — 强一致性事务更适合分层/单体架构，事件驱动天然是最终一致性

3. 微内核架构中，插件与核心系统之间的通信通常通过什么机制？

  • A. HTTP REST 调用
  • B. 预定义的插件接口（API）
  • C. 共享内存
  • D. 数据库
答案B — 核心系统定义插件 API，所有插件通过该接口与核心交互

4. 分层架构中，"跨层调用"违反了哪种分层策略？

  • A. 宽松分层
  • B. 严格分层
  • C. 所有分层策略
  • D. 跨层调用在任何分层中都是允许的
答案B — 严格分层只允许调用直接下层，不允许跨层

5. 事件驱动架构中，事件的"发布-订阅"模式相比"点对点"模式的优势是？

  • A. 更高的可靠性
  • B. 发布者和订阅者完全解耦
  • C. 更低的消息延迟
  • D. 更强的消息顺序保证
答案B — 发布者不知道谁订阅了事件，订阅者不知道谁发布了事件，完全解耦

6. 关于微内核架构（插件架构），以下描述错误的是？

  • A. 核心系统提供最小功能集
  • B. 插件之间可以直接通信
  • C. 新增功能只需开发新插件
  • D. Eclipse IDE 是微内核架构的典型案例
答案B — 插件之间通常通过核心系统间接通信，不应直接耦合

7. 管道-过滤器风格中，关于"过滤器"的设计要求，正确的是？

  • A. 每个过滤器必须知道上游和下游过滤器
  • B. 过滤器之间必须使用相同的数据格式
  • C. 每个过滤器是独立的、可复用的处理单元
  • D. 过滤器不能并发执行
答案C — 过滤器不依赖其他过滤器，只关心输入/输出接口

8. SOA 中，服务提供者（Provider）、服务消费者（Consumer）、服务注册中心（Registry）三者的关系是？

  • A. Provider 直接调用 Consumer
  • B. Provider 先注册到 Registry，Consumer 从 Registry 发现 Provider 后再调用
  • C. Consumer 和 Provider 必须同时在线
  • D. Registry 负责转发所有服务调用
答案B — Provider 注册服务到 Registry，Consumer 查询 Registry 获取 Provider 地址，然后直连调用

9. 事件驱动架构中，"事件溯源"（Event Sourcing）的核心思想是？

  • A. 记录事件的发送者
  • B. 将所有状态变更存储为事件序列，而非存储最终状态
  • C. 对事件进行加密存储
  • D. 使用事件日志做性能监控
答案B — 事件溯源不存储当前状态，而是存储所有状态变更事件，通过重放事件重建当前状态

10. 以下哪种场景最适合使用"微内核架构"？

  • A. 需要处理实时股票行情的交易系统
  • B. 需要支持大量第三方扩展的 IDE 或 CMS 平台
  • C. 需要保证 ACID 事务的银行转账系统
  • D. 需要流式处理的日志分析平台
答案B — 微内核架构的核心优势就是通过插件机制支持灵活扩展

11. 分层架构中，关于"依赖倒置"的应用，以下描述正确的是？

  • A. 上层定义接口，下层实现接口，上层依赖接口而非实现
  • B. 下层定义接口，上层实现接口
  • C. 分层架构不支持依赖倒置
  • D. 依赖倒置意味着取消所有分层
答案A — 通过在上层定义接口、下层实现，使上层不直接依赖下层实现细节

12. SOA 与微服务的最本质区别是？

  • A. SOA 使用 HTTP，微服务使用 gRPC
  • B. SOA 以 ESB 为中心、强调复用；微服务去中心化、强调独立部署
  • C. SOA 只能用于 Java，微服务语言无关
  • D. SOA 比微服务更新
答案B — SOA 的核心是 ESB 作为集中式集成枢纽，微服务的核心是去中心化的轻量通信

13. 管道-过滤器风格中，"Tee"（三通）过滤器的作用是？

  • A. 过滤掉不符合条件的数据
  • B. 将数据同时分发到多个下游管道
  • C. 合并多个上游管道的数据
  • D. 对数据进行排序
答案B — Tee 过滤器将同一份数据复制并发送到多个下游过滤器（如：同时做日志记录和业务处理）

14. 某系统使用事件驱动架构，当订单创建事件发布后，库存服务和通知服务都订阅了该事件。如果库存服务处理失败，以下哪种异常处理策略最合理？

  • A. 回滚订单创建
  • B. 库存服务进行重试，通知服务正常发送通知
  • C. 通知服务也回滚
  • D. 不做任何处理
答案B — 事件驱动架构中，各订阅者独立处理事件，一个消费者的失败不应影响其他消费者。库存服务自身做补偿重试，通知服务可正常执行。

15. 关于微内核架构中"核心系统"的设计原则，错误的是？

  • A. 核心系统应尽量精简
  • B. 核心系统定义插件接口规范
  • C. 核心系统应包含所有业务逻辑
  • D. 核心系统管理插件生命周期
答案C — 核心系统只提供最小功能集和插件管理，业务逻辑应尽量在插件中实现

多选题（4 题）

16.（多选）分层架构的常见层次包括？

  • A. 表示层（Presentation）
  • B. 业务逻辑层（Business Logic）
  • C. 数据访问层（Data Access）
  • D. 消息队列层
  • E. 数据库层
答案A、B、C、E — 经典四层：表示层、业务逻辑层、数据访问层、数据库层。消息队列属于基础设施，不是独立层。

17.（多选）关于 SOA 和微服务的对比，正确的有？

  • A. SOA 通常使用 ESB 作为集成中枢
  • B. 微服务强调"去中心化"的数据管理
  • C. SOA 的服务粒度通常比微服务粗
  • D. 微服务必须使用容器化部署
  • E. SOA 的服务契约通常是标准化的（WSDL）
答案A、B、C、E — D 错误：微服务可以用容器、虚拟机、甚至物理机部署，容器化不是必须的

18.（多选）事件驱动架构的挑战包括？

  • A. 调试难度高（事件流追踪困难）
  • B. 最终一致性（不是强一致性）
  • C. 事件版本兼容性
  • D. 无法实现异步处理
  • E. 事件风暴（过多事件导致系统混乱）
答案A、B、C、E — D 错误：事件驱动天然支持异步处理，这是它的优势

19.（多选）管道-过滤器风格中的过滤器可以有哪些类型？

  • A. 生产者（Source）— 产生数据
  • B. 转换器（Transformer）— 转换数据格式
  • C. 测试器（Tester）— 过滤数据
  • D. 消费者（Sink）— 接收最终数据
  • E. 协调器（Orchestrator）— 管理过滤器执行顺序
答案A、B、C、D — 过滤器按角色分为生产者、转换器、测试器、消费者。协调/编排不改变数据本身，通常由管道或外部框架完成。

20. 请画出分层架构的 4 层经典结构图，并说明每层的典型职责和常见技术栈。

参考

21. SOA 中 ESB 的主要功能是什么？为什么说 ESB 可能成为瓶颈？

参考

ESB 五功能：消息路由、协议转换、数据转换、服务编排、安全/监控。

成为瓶颈的三大原因：

  1. 单点故障：所有服务通信经过 ESB，ESB 宕机=整个 SOA 瘫痪
  2. 性能瓶颈：ESB 需要解析和转换每一笔消息，高并发时成为吞吐量天花板
  3. 复杂度集中：所有集成逻辑挤在 ESB 中→变更风险高、维护成本大

微服务正是为了解决 ESB 集中化问题而兴起的。

22. 对比事件驱动架构中的"发布-订阅"和"事件流"（Event Streaming）两种模式的核心差异。

参考
维度发布-订阅事件流
消息保留消费即删除持久化存储，可回溯
消费方式实时推送给所有订阅者消费者主动拉取（Pull）

23. 某视频处理平台需要：原始视频 → 格式检测 → 转码 → 加水印 → 缩略图生成 → CDN分发。建议什么架构风格？为什么？若追加"视频审核"步骤（人工+AI），对架构有什么影响？

参考

推荐管道-过滤器风格：

  • 每步封装为独立过滤器，可复用可替换
  • 流程可灵活编排（不同视频类型走不同处理管线）
  • 各步骤可并行处理不同视频

追加审核步骤的影响：

  • 在管道中插入"审核过滤器"（AI自动审核 → 人工审核 → 通过/拒绝）
  • 若审核是异步的（人工介入），需要在审核过滤器处暂停管线，使用事件驱动机制通知审核结果
  • 本质上是"管道-过滤器 + 事件驱动"的混合架构

24. 某大型电商系统，订单模块需要高一致性（不能丢单），推荐模块需要高灵活性（频繁策略变更）。如果你只能选一种架构风格来设计整个系统，会有什么问题？你会如何解决？

参考

不能"一刀切"：

问题若全用分层若全用事件驱动若全用微服务
订单模块✅ 事务简单❌ 最终一致性不可接受✅ 可保证一致性
推荐模块❌ 耦合严重✅ 灵活解耦✅ 独立迭代

解决方案：架构风格混合使用

  • 订单核心：分层架构（Repository 模式），保证 ACID 事务
  • 推荐引擎：事件驱动，监听用户行为事件，灵活更新推荐策略
  • 接口层：统一 API 网关，对外暴露一致接口

关键约束：订单模块的领域事件发布到消息队列，推荐模块异步消费（最终一致性可接受）。

25. 设计一个在线考试系统，需求如下：

  • 考生答题时答案实时保存（防止断网丢失）
  • 考试结束后 10 秒内出成绩
  • 支持单选题、多选题、填空题、编程题
  • 编程题的判题由第三方判题服务完成（判题可能需要 1~5 分钟）

请为该系统选择合适的架构风格组合，并说明每个子系统的风格选型理由。

参考
子系统架构风格理由
答题保存事件驱动每次输入触发保存事件，异步写入（断网→本地队列→联网后重放）
成绩计算管道-过滤器客观题（即时判分）→ 编程题（等待判题回调）→ 合并成绩 → 输出
试卷配置分层架构CURD 操作，无复杂交互，分层足够
编程题判题仓库（Repository）判题服务异步返回结果，写入共享的判题结果存储，成绩模块读取

整体是事件驱动 + 管道-过滤器 + 分层的异质混合架构。
