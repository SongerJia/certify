## 二、ATAM 概述

### 2.1 全称与目的

| 项目 | 内容 |
|:---|:---|
| **全称** | Architecture Tradeoff Analysis Method |
| **中文** | 架构权衡分析方法 |
| **提出者** | SEI (Software Engineering Institute, CMU) |
| **核心目的** | 在多个质量属性之间找到**最佳权衡** |
| **输入** | 架构设计方案 + 质量属性需求 |
| **输出** | 风险列表 + 权衡点分析 + 改进建议 |

### 2.2 为什么要用 ATAM？

> 单一质量属性最优通常意味着其他质量属性的牺牲。ATAM 帮助我们**显式地看到这些权衡**。

```
增加冗余 → 提高可用性 ✓ → 增加成本/复杂度 ✗
引入缓存 → 提高性能 ✓ → 增加一致性风险 ✗
严格分层 → 提高可维护性 ✓ → 降低性能 ✗
```

### 2.3 ATAM 的核心四问

1. 架构方案能否满足关键质量属性需求？
2. 在满足这些需求的过程中，有哪些**风险**？
3. 不同的质量属性之间存在哪些**权衡**？
4. 对这些权衡，各方能否达成**共识**？

## 三、ATAM 九步法

```
Phase 1: 准备与评审
  Step 1: 介绍 ATAM               ← 方法论介绍
  Step 2: 介绍业务驱动因素         ← 业务目标→质量属性
  Step 3: 介绍架构                ← 架构师讲解方案

Phase 2: 调查与分析
  Step 4: 确定架构方法             ← 识别采用的架构风格/模式
  Step 5: 生成质量属性效用树       ← ★ 核心步骤
  Step 6: 分析架构方法             ← 针对高优先级场景分析

Phase 3: 测试与报告
  Step 7: 头脑风暴与优先级排序     ← 利益相关者补充场景
  Step 8: 再次分析架构方法         ← 验证新场景
  Step 9: 提交报告                ← 汇总风险/权衡/建议
```

## 四、核心步骤详解

### Step 5：质量属性效用树（最核心，必考）

#### 效用树结构

```
        根节点 (Utility)
              │
    ┌─────────┼─────────┐
    │         │         │
 可用性     性能     安全性     ← 质量属性
    │         │         │
 ┌──┴──┐  ┌──┴──┐  ┌──┴──┐
 │故障 │  │响应 │  │认证 │     ← 质量属性求精
 │恢复 │  │时间 │  │授权 │
 └──┬──┘  └──┬──┘  └──┬──┘
    │        │        │
 ┌──┴───────┴────────┴──┐
 │  具体场景（带优先级）    │     ← 场景
 │  (H,H), (H,M), (M,L) │
 └──────────────────────┘
```

#### 效用树中场景的优先级标记

| 标记 | 含义 | 处理方式 |
|:---|:---|:---|
| **(H, H)** | 高重要性 + 高难度 | 重点分析 |
| **(H, M)** | 高重要性 + 中等难度 | 重点分析 |
| **(M, H)** | 中等重要性 + 高难度 | 次优先级 |
| **(L, \*)** | 低重要性 | 暂缓分析 |

#### 效用树构建步骤

1. **确定根节点**：Utility（效用）
2. **列出质量属性**：可用性、性能、安全性、可修改性...
3. **质量属性求精**：将抽象属性细化为具体子属性
4. **生成场景**：为每个子属性编写具体可测场景
5. **标注优先级**：利益相关者投票 (H/M/L, H/M/L)

### 效用树实例（电商系统）

```
Utility
├── 可用性
│   ├── 故障检测
│   │   └── (H,H) 主库故障后 30 秒内检测到并告警
│   ├── 故障恢复
│   │   └── (H,H) 主备切换 60 秒内完成，数据零丢失
│   └── 降级处理
│       └── (H,M) 秒杀期间非核心服务降级
├── 性能
│   ├── 响应时间
│   │   └── (H,H) 正常负载下 95% 请求 &lt; 200ms
│   ├── 吞吐量
│   │   └── (H,H) 支持 10000 TPS 下单峰值
│   └── 并发
│       └── (H,M) 100 万用户同时在线
├── 安全性
│   ├── 认证
│   │   └── (H,H) 支持 OAuth2.0 + 多因素认证
│   └── 数据保护
│       └── (H,H) 用户敏感数据 AES-256 加密存储
└── 可修改性
    └── 变更范围
        └── (M,H) 新支付渠道接入 &lt; 3 人日
```

## 五、风险/非风险/敏感点/权衡点

### 5.1 四个核心概念

| 概念 | 英文 | 定义 | 例子 |
|:---|:---|:---|:---|
| **风险点** | Risk | 可能导致问题的架构决策 | "单点数据库可能成为性能瓶颈" |
| **非风险点** | Non-Risk | 已确认没有问题的地方 | "采用标准 HTTP 协议无技术风险" |
| **敏感点** | Sensitivity Point | 某个属性的微小变化会**大幅影响**系统质量 | "缓存命中率每降低 1%，响应时间增加 20ms" |
| **权衡点** | Tradeoff Point | 一个决策的优化会**损害**另一个质量属性 | "加密强度提高→安全性↑，但性能↓" |

### 5.2 四者关系

```
敏感点 ── 描述某个质量属性对变化的敏感程度
   │
权衡点 ── 当敏感点涉及两个或多个质量属性时
   │
风险点 ── 当架构决策可能无法满足质量属性需求
   │
非风险点 ── 已确认无风险的设计决策
```

### 5.3 考试常见考法

**案例题**：给出架构描述，要求：
1. 识别至少 3 个**风险点**
2. 识别至少 1 个**敏感点**
3. 识别至少 1 个**权衡点**

> **技巧**：风险点 = "可能出问题"；敏感点 = "小变化大影响"；权衡点 = "鱼与熊掌"

## 六、ATAM 产出物模板

### ATAM 评估报告结构

```
1. 评估概述
   - 评估目的
   - 参与人员（评估团队 + 利益相关者）
   - 评估日期

2. 业务驱动因素
   - 核心业务目标
   - 关键质量属性需求（优先级排序）

3. 架构方案概述
   - 采用的架构风格/模式
   - 关键设计决策

4. 效用树
   - 质量属性 → 场景 → 优先级

5. 风险分析
   - 风险点 (Risks)
   - 非风险点 (Non-Risks)
   - 敏感点 (Sensitivity Points)
   - 权衡点 (Tradeoff Points)

6. 改进建议
   - 风险缓解策略
   - 替代方案建议
```

## 七、对比：ATAM vs SAAM

| 维度 | ATAM | SAAM |
|:---|:---|:---|
| **全称** | Architecture Tradeoff Analysis Method | Software Architecture Analysis Method |
| **核心关注** | 多质量属性间的**权衡** | 单个功能/质量属性的**评估** |
| **产出核心** | 权衡点、风险点 | 场景交互影响的评估 |
| **复杂度** | 高（9 步） | 中（6 步） |
| **适用场景** | 重大架构决策评审 | 快速架构评估 |
| **是否需要效用树** | ✅ | ❌ |

> **一句话区分**：ATAM = "这怎么权衡？"，SAAM = "这能满足需求吗？"

### 简答题（3 题）

**20.** 某系统引入 Redis 缓存后，性能提升了 50%，但缓存与数据库的不一致窗口从 0 变为 5 秒。如果进一步发现"缓存命中率降低 1%，响应时间增加 20ms"，这两个现象分别属于哪种分析点？为什么？

<details><summary>参考</summary>

| 现象 | 分析点类型 | 理由 |
|:---|:---|:---|
| 性能↑50% & 一致性↓ | **权衡点** | 两个质量属性相互影响，一个提升以另一个下降为代价 |
| 命中率→响应时间 | **敏感点** | 只涉及性能一个质量属性，响应时间对命中率非常敏感 |

区分标准：**涉及 1 个质量属性 = 敏感点，涉及 ≥2 个质量属性且互相制约 = 权衡点**。
</details>

**21.** 请列出 ATAM 的 9 个步骤，并给每步写一句话的简要说明。

<details><summary>参考</summary>

| 步骤 | 名称 | 一句话说明 |
|:---:|:---|:---|
| 1 | 展示 ATAM | 向利益相关者介绍评估方法 |
| 2 | 描述业务目标 | 明确系统的核心业务驱动力 |
| 3 | 描述架构 | 架构师展示当前架构方案 |
| 4 | 识别架构方法 | 归纳架构中使用的关键设计决策 |
| 5 | 生成效用树 | 构建质量属性→场景→优先级的树 |
| 6 | 分析架构方法 | 针对高优先级场景逐一分析架构能否满足 |
| 7 | 头脑风暴 + 场景优先级 | Phase 2：全体利益相关者贡献场景 |
| 8 | 分析架构方法（第二轮） | 对新场景做第二遍深入分析 |
| 9 | 呈现结果 | 汇总风险/非风险/敏感/权衡点，形成报告 |

</details>

**22.** 如果要在 1 天内完成一个微服务系统的 ATAM 快速评估（而非完整 2 天的标准流程），你会如何裁剪步骤？哪些步骤不能省略？

<details><summary>参考</summary>

**可裁剪的**：
- 步骤 1：5 分钟快速介绍（而不是 1 小时）
- 步骤 7：头脑风暴改为预先收集的 Top 10 场景（而非开放式讨论）
- 步骤 8：仅分析 Top 5 高优先级场景

**不能省略的**：
- 步骤 3（描述架构）— 没有架构描述就没法评估
- 步骤 5（效用树）— ATAM 的核心工具
- 步骤 6（分析架构方法）— 评估的本质
- 步骤 9（呈现结果）— 没有结论的评估是无效的

**1 天压缩版流程**：
上午：步骤 1(5min) → 2(30min) → 3(60min) → 4(30min) → 5(60min)
下午：步骤 6(90min) → 预设场景分析(60min) → 9(30min)

</details>

### 情景分析（3 题）

**23.** 作为架构师，评估两个候选方案：方案 A 采用单体架构（开发快，扩展性差）；方案 B 采用微服务（扩展性好，运维复杂）。请用 ATAM 思维回答：
1. 需要收集什么信息？
2. 构建怎样的效用树？
3. 如何得出评估结论？

<details><summary>参考</summary>

**1. 需要收集的信息**：
- 业务目标：上市时间（Time to Market）vs 长期扩展性
- 团队信息：人数、技能分布、是否有微服务经验
- 技术约束：部署环境、运维能力、基础设施

**2. 效用树**：
```
系统效用
├── 可修改性/新功能上线周期/(H,?)
├── 可扩展性/支持未来N倍用户/(H,?)
├── 开发效率/初始版本交付/(H,?)
├── 运维复杂度/日常人力投入/(M,?)
└── 可用性/服务故障恢复/(M,?)
```

**3. 评估结论**：
- 方案A：开发效率(H,L) ✅ | 扩展性(H,H) ❌ → **风险点：后期扩展需重构**
- 方案B：扩展性(H,M) ✅ | 开发效率(H,H) ❌ → **风险点：初期交付延迟**
- **权衡点**：开发速度 vs 长期可维护性
- 若业务处于验证期 → 选 A（快速验证后重构）
- 若业务稳定增长 → 选 B（一步到位避免未来重构成本）

</details>

**24.** 为以下效用树片段中的场景标注 (重要性, 难度)，并指出哪个场景应该最先分析：

- 场景 1：系统故障后 30 秒内自动恢复
- 场景 2：新增一个数据导出功能，开发周期 2 天
- 场景 3：支持从 MySQL 迁移到 PostgreSQL，业务无感知
- 场景 4：DDoS 攻击下核心服务不中断

<details><summary>参考</summary>

| 场景 | 重要性 | 难度 | 标注 | 优先级 |
|:---|:---|:---|:---|:---:|
| 1 | 高 | 高 | (H,H) | **1st** |
| 2 | 中 | 低 | (M,L) | 4th |
| 3 | 低 | 高 | (L,H) | 3rd |
| 4 | 高 | 高 | (H,H) | **2nd** |

(H,H) 场景应优先分析——它们是最可能成为架构瓶颈的地方。场景 1（可用性）> 场景 4（安全）因为故障恢复是更常见的架构问题。场景 3 (L,H) 次之（难但没那么重要）。场景 2 (M,L) 最简单，最后分析或不需深入分析。

</details>

**25.** 设计一个"在线医疗预约系统"的 ATAM 效用树。要求：
1. 至少包含 3 个质量属性
2. 每个质量属性下至少 2 个关注点（二级节点）
3. 每个关注点下至少 1 个场景叶子
4. 标注所有场景的优先级 (重要性, 难度)

<details><summary>参考</summary>

```
在线医疗预约系统 效用树

├── (H) 可用性
│   ├── 故障恢复
│   │   ├── (H,H) 数据库主库宕机 → 30s 内切换到备库，不丢失预约数据
│   │   └── (M,H) 第三方支付服务超时 → 自动重试 3 次，失败后降级为线下支付
│   └── 服务降级
│       └── (H,M) 挂号高峰期 → 排队机制，HTTP 200 + 排队序号，而非 503
│
├── (H) 安全性
│   ├── 数据保护
│   │   ├── (H,H) 患者隐私数据 → 传输和存储全链路 AES-256 加密
│   │   └── (H,M) 处方信息 → 只有主治医生和患者本人可查看
│   └── 访问控制
│       └── (M,M) 医生不能查看非挂号患者的病历 → RBAC 权限控制
│
├── (H) 性能
│   ├── 并发挂号
│   │   └── (H,H) 热门专家放号时 → 1000 人同时抢号，响应 < 2s，无超卖
│   └── 查询响应
│       └── (M,L) 患者查询历史病历 → < 1s 返回最近 3 年记录
│
└── (M) 可修改性
    ├── 排班规则
    │   └── (M,M) 节假日排班规则变更 → 修改排班引擎配置，不修改代码
    └── 支付渠道
        └── (L,L) 新增医保支付 → 新增支付插件，< 3 人天
```

**优先分析顺序**：(H,H) 场景 → 热门专家抢号、主库故障恢复、患者数据加密。

</details>
