## 二、核心概念

### 2.1 什么是软件架构 —— 三种经典定义

| 来源 | 定义 | 关键视角 |
|:---|:---|:---|
| **IEEE 1471-2000** | 系统的基本组织，体现在其组件、组件之间的关系、组件与环境的关系，以及指导其设计和演化的原则 | 静态结构 |
| **Bass, Clements, Kazman** | 软件架构是一个或多个结构，包含软件元素、元素的外部可见属性以及元素之间的关系 | 运行时行为 |
| **Garlan & Shaw** | 架构 = 组件 + 连接件 + 约束 | 形式化描述 |

> **一句话记忆**：架构回答三个问题 —— "系统由哪些部分组成？它们怎么交互？为什么这样设计？"

### 2.2 架构风格 ≠ 架构模式 ≠ 设计模式

| 层次 | 粒度 | 例子 | 作用域 |
|:---|:---|:---|:---|
| **架构风格** | 系统级 | 分层、事件驱动、微服务 | 整个系统的组织结构 |
| **架构模式** | 子系统级 | MVC、Broker、Pipe-Filter | 特定子问题的解决方案 |
| **设计模式** | 组件级 | 工厂、单例、观察者 | 单个组件的内部设计 |

```
架构风格（宏观组织）
  └── 架构模式（中观结构）
        └── 设计模式（微观实现）
```

### 2.3 架构风格的六要素（考试常考填空题）

> **记忆口诀**：**组连属约模原** — 组件、连接、属性、约束、模型、原理

| 要素 | 含义 | 例子（以分层架构为例） |
|:---|:---|:---|
| **组件 (Component)** | 构成系统的基本计算单元 | 表示层、业务层、数据层 |
| **连接件 (Connector)** | 组件之间的交互机制 | 方法调用、HTTP 请求 |
| **属性 (Property)** | 组件的特征描述 | 层内高内聚，层间低耦合 |
| **约束 (Constraint)** | 组件连接的规则 | 上层调用下层，下层不能调用上层 |
| **模型 (Model)** | 风格的抽象结构表示 | UML 层次图 |
| **原理 (Rationale)** | 为什么这样设计 | 关注点分离，隔离变化 |

### 2.4 架构抉择的核心权衡

每种架构风格的选择，本质上是在**不同质量属性**之间做权衡：

```
性能 ←→ 安全性
    ↕      ↕
灵活性 ←→ 可维护性
    ↕      ↕
开发效率 ←→ 运行效率
```

> **考试要点**：选择题经常给一个场景，问你"应该选什么架构风格，为什么"。牢记：**没有最好的风格，只有最合适的权衡**。

## 三、7 大类架构风格总览

> **记忆方法**：按数据流、调用/返回、独立组件、虚拟机、仓库、C2、特定领域 来分组记忆

### 第一类：数据流风格（Data Flow）

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **管道-过滤器** | 数据流经一系列处理单元 | 顺序处理，过滤器可复用 |
| **批处理** | 数据成批处理 | 非实时，定时触发 |

### 第二类：调用/返回风格（Call/Return）

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **主程序/子程序** | 功能分解为树状调用 | 经典结构化编程 |
| **分层架构** | 按抽象层次组织 | 下层为上层服务 |
| **面向对象** | 对象封装数据+操作 | 继承、多态 |
| **客户端/服务器** | Client 请求，Server 响应 | 分布式交互 |

### 第三类：独立组件风格（Independent Components）

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **事件驱动** | 事件触发组件行为 | 松耦合、异步 |
| **进程通信** | 独立进程间消息传递 | 高并发 |

### 第四类：虚拟机风格（Virtual Machine）

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **解释器** | 逐条解释执行指令 | 灵活，性能较低 |
| **规则引擎** | 规则库驱动推理 | 业务逻辑外置 |

### 第五类：仓库风格（Repository）

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **黑板系统** | 多个组件共享全局数据 | 适合 AI/信号处理 |
| **数据库为中心** | 以 DB 为共享状态 | 经典企业应用 |

### 第六类：C2 风格

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **C2 架构** | 组件通过连接器上下通信 | 请求向上，通知向下 |

### 第七类：特定领域风格

| 风格 | 核心思想 | 特点 |
|:---|:---|:---|
| **SOA** | 服务化，ESB 集成 | 企业级松耦合 |
| **微服务** | 小粒度独立服务 | 去中心化治理 |
| **MVC/MVP/MVVM** | UI 层关注点分离 | 前端架构 |

## 四、考试重点聚焦

### 选择题高频考点

1. **给定场景选风格**：如"需要流式数据处理"→管道-过滤器
2. **风格归类**：某风格属于哪一大类（如 SOA 属于独立组件风格）
3. **风格 vs 模式区分**：MVC 是架构模式，不是架构风格
4. **要素填空**：架构风格的六要素

### 案例题常见考法

- 给一个系统描述，问：当前用了什么架构风格？有什么优缺点？
- 要求根据质量属性需求，为新系统选择合适的架构风格并说明理由

### 论文关联

- 微服务架构设计与实践
- 分层架构在企业系统中的应用

**21.** 某系统要求极高的可扩展性，但也要保证核心模块的稳定性。你如何将不同架构风格混合使用？请举例说明。

<details><summary>参考</summary>

采用**架构风格的"异质混合"**策略：
- 核心交易模块（支付、账务）：**分层架构**（稳定性优先，严格依赖管理）
- 业务扩展模块（营销、推荐）：**微服务架构**（独立部署，快速迭代）
- 实时数据处理（日志、监控）：**管道-过滤器 + 事件驱动**（流式处理，松耦合）

关键原则：每种风格应用在其最擅长的子系统上，接口层面用统一的协议（REST/消息队列）连接。
</details>

</details>

### 情景分析（3 题）

**23.** 某电商平台需要处理多种数据源的实时数据融合：订单数据（数据库）、用户行为（埋点）、库存变动（MQ）。各处理逻辑各异且频繁变化。请完成以下分析：

- 建议使用什么架构风格？为什么？
- 若新增"物流数据"源，需要修改系统哪些部分？

<details><summary>参考</summary>

建议使用**管道-过滤器风格**，因为：
1. 每种数据源处理可封装为独立过滤器（订单解析、行为聚合、库存过滤）
2. 新增物流数据源只需新增一个过滤器并接入管道
3. 不同过滤器的处理逻辑可以独立修改和部署
4. 可以并行处理不同数据源

新增物流数据源时，只需：①实现物流数据过滤器；②将其连接到输入管道；③注册到过滤器编排器。其他过滤器无需修改。
</details>

**24.** 某团队在架构评审中，提出了两个方案：
- 方案 A：全部采用微服务架构
- 方案 B：核心系统用分层架构，新功能用微服务

作为架构师，你会选哪个？从六要素的角度阐述理由。

<details><summary>参考</summary>

选择**方案 B**（异质混合），从六要素分析：

| 六要素 | 方案 B 优势 |
|:---|:---|
| **组件** | 核心与扩展组件职责明确，边界清晰 |
| **连接件** | 核心内部用高效调用，扩展间用轻量 REST/MQ |
| **约束** | 分层约束保证核心稳定，微服务约束保证扩展灵活 |
| **模型** | 不同部分建模粒度匹配其需求 |
| **配置** | 避免了"一刀切"的架构配置 |
| **推理** | 稳定性+扩展性两个目标分别满足，而不是折中 |

</details>

**25.** 某系统需要处理多个异构数据源的数据融合，各数据源的处理逻辑不同且可能变化。你建议使用什么架构风格？如果数据源之间还需要做关联计算（如订单数据和支付数据的交叉验证），原来的风格还适合吗？

<details><summary>参考</summary>

基础场景使用**管道-过滤器**（原因同前）。

如果需要关联计算，建议在管道-过滤器基础上增加**仓库/黑板**组件：
- 各过滤器将处理后的标准数据写入共享数据存储
- 关联计算模块从共享存储中读取并做交叉验证
- 本质上是"管道-过滤器 + 仓库"的混合架构

这种混合策略既保留了数据流处理的灵活性，又解决了跨数据源关联计算的问题。
