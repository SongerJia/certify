# 08 — 架构文档化与 4+1 视图

> **对应计划**：第 2 周 周五 | **难度**：中等 | **考试权重**：综合 2~4 分 + 案例偶尔涉及

---

## 一、学习目标

- [ ] 掌握 4+1 视图模型的 5 个视角及其关注点
- [ ] 理解每个视图对应的 UML 图类型
- [ ] 能区分逻辑视图和开发视图（考试最常见混淆点）
- [ ] 了解架构文档化的最佳实践

---

## 二、为什么需要架构文档化？

> 架构不只是画几张图。好的架构文档让**不同角色的人能从各自视角理解系统**。

| 角色 | 关心的问题 | 对应的视图 |
|:---|:---|:---|
| **最终用户** | 系统做什么？有哪些功能？ | 场景视图 |
| **系统集成师** | 系统如何处理并发？怎么部署？ | 进程视图 + 物理视图 |
| **开发人员** | 代码怎么组织？包依赖是什么？ | 开发视图 + 逻辑视图 |
| **项目经理** | 开发任务怎么划分？进度如何？ | 开发视图 + 场景视图 |
| **运维人员** | 部署到哪些机器？网络拓扑？ | 物理视图 |

## 三、4+1 视图模型

> **Philippe Kruchten (1995)** 提出，是 IEEE 1471 的基础，被 RUP (Rational Unified Process) 采用。

### 3.1 五视图总览

```
                    ┌─────────────────┐
                    │   场景视图 (+1)   │
                    │  (用例/需求驱动)  │
                    └────────┬────────┘
                             │
        ┌────────┬───────────┼───────────┬────────┐
        ↓        ↓           ↓           ↓        ↓
   ┌─────────┐ ┌────────┐ ┌────────┐ ┌────────┐
   │逻辑视图  │ │进程视图 │ │开发视图 │ │物理视图 │
   │(Logical)│ │(Process)│ │(Devel.) │ │(Physical)│
   └─────────┘ └────────┘ └────────┘ └────────┘
     功能需求     非功能      软件管理    系统拓扑
     (做什么)   (怎么跑)    (怎么拆)    (部署到哪)
```

### 3.2 各视图详解

#### ① 逻辑视图 (Logical View) — 给"设计师"看

| 维度 | 内容 |
|:---|:---|
| **关注点** | 系统提供的**功能需求** |
| **受众** | 最终用户、业务分析师 |
| **核心问题** | "系统做什么？" |
| **结构** | 类、包、子系统 |
| **UML 图** | 类图、对象图、状态图、交互图 |

**例子**：电商系统的逻辑视图关注的是"商品、订单、用户、支付"这些**业务概念**及其关系。

#### ② 进程视图 (Process View) — 给"系统集成师"看

| 维度 | 内容 |
|:---|:---|
| **关注点** | 系统的**运行时行为** |
| **受众** | 系统集成师、性能工程师 |
| **核心问题** | "系统怎么跑？" |
| **结构** | 进程、线程、任务 |
| **UML 图** | 活动图、时序图、通信图 |

**关键非功能需求**：
- 并发性 (Concurrency)
- 可用性 (Availability)
- 性能 (Performance)
- 可扩展性 (Scalability)
- 完整性 (Integrity)

**例子**：电商系统的进程视图关注的是"订单服务部署为 3 个实例，通过 Nginx 负载均衡，支付回调走消息队列异步处理"。

#### ③ 开发视图 (Development View) — 给"程序员"看

| 维度 | 内容 |
|:---|:---|
| **关注点** | 软件的**静态组织结构** |
| **受众** | 开发人员、项目管理者 |
| **核心问题** | "代码怎么组织？" |
| **结构** | 模块、子系统、层、库 |
| **UML 图** | 包图、组件图 |

**原则**：
- 高内聚、低耦合
- 分层清晰
- 便于并行开发

**例子**：电商系统的开发视图关注的是"项目分为 order-module、payment-module、product-module 三个 Maven 子模块"。

#### ④ 物理视图 (Physical View) — 给"运维工程师"看

| 维度 | 内容 |
|:---|:---|
| **关注点** | 系统在**物理硬件**上的分布 |
| **受众** | 运维工程师、网络工程师 |
| **核心问题** | "系统部署在哪里？" |
| **结构** | 节点、网络连接 |
| **UML 图** | 部署图 |

**关注的非功能需求**：
- 可用性（多机房、多副本）
- 可靠性（容错）
- 性能（CDN、就近访问）
- 可扩展性（横向扩容）

**例子**：电商系统的物理视图描述"Web 层 3 台服务器、应用层 5 台服务器、数据库主从 2 台 + Redis 集群 3 台"的物理部署拓扑。

#### ⑤ 场景视图 (+1) — 把四个视图串起来

| 维度 | 内容 |
|:---|:---|
| **关注点** | 系统关键用例的**端到端流程** |
| **受众** | 所有利益相关者 |
| **核心问题** | "各视图如何协同工作？" |
| **结构** | 用例、场景 |
| **UML 图** | 用例图 |

> **为什么叫 +1**：场景视图是"粘合剂"，它用几个关键场景（用例）把逻辑视图、进程视图、开发视图、物理视图串联起来，验证架构的一致性。

## 四、4+1 视图速查表

| 视图 | 重点 | 受众 | 核心 UML 图 | 关键词 |
|:---|:---|:---|:---|:---|
| **逻辑视图** | 功能 | 用户/BA | 类图 | 做什么 |
| **进程视图** | 运行时 | 集成师 | 时序图/活动图 | 怎么跑 |
| **开发视图** | 代码组织 | 程序员 | 包图/组件图 | 怎么写 |
| **物理视图** | 部署 | 运维 | 部署图 | 放哪里 |
| **场景视图** | 贯穿 | 所有人 | 用例图 | 串起来 |

> **记忆口诀**：**逻功进运开代物部场串** — 逻辑看功能、进程看运行、开发看代码、物理看部署、场景串一切。

---

## 五、常见混淆辨析

### 5.1 逻辑视图 vs 开发视图（考试最高频混淆）

| 对比维度 | 逻辑视图 | 开发视图 |
|:---|:---|:---|
| 关注 | 业务概念（领域模型） | 代码组织（模块结构） |
| 回答了 | "系统有什么功能？" | "代码分包分模块？" |
| 变化原因 | 业务需求变化 | 团队组织变化/技术选型 |
| 关系 | 一个逻辑组件可能横跨多个开发模块 | 一个开发模块可能包含多个逻辑组件 |
| UML | 类图（领域类） | 包图/组件图 |

> **一句话区分**：逻辑视图画的是"Order 类有哪些属性和方法"，开发视图画的是"order.jar 依赖 payment.jar"。

### 5.2 进程视图 vs 物理视图

| 对比维度 | 进程视图 | 物理视图 |
|:---|:---|:---|
| 关注 | 进程/线程的**动态交互** | 硬件的**静态拓扑** |
| 回答 | "服务怎么通信？" | "服务器怎么连接？" |
| UML | 时序图 | 部署图 |

### 5.3 什么时候用哪个视图？

```
回答 "新功能怎么实现？"         → 逻辑视图
回答 "能不能抗住 10 万 QPS？"  → 进程视图
回答 "代码放哪个 Maven 模块？" → 开发视图
回答 "部署到哪些机器上？"       → 物理视图
回答 "用户下单怎么走通全流程？" → 场景视图
```

## 六、架构文档化最佳实践

### 6.1 文档化规则

| 规则 | 说明 |
|:---|:---|
| **写给读者看** | 不同读者需要不同详细程度 |
| **先结构后细节** | 从 4+1 视图总览开始 |
| **保持一致性** | 不同视图描述的是同一个系统 |
| **记录设计决策和理由** | 不只记录"是什么"，还要记录"为什么" |
| **文档即代码般维护** | 架构变化时同步更新文档 |

### 6.2 反模式（不要这样做）

| 反模式 | 问题 |
|:---|:---|
| **空中楼阁** | 文档和实际代码完全不同步 |
| **统一视图** | 只有一个"万能视图"（通常是类图） |
| **过度详细** | 记录所有细节，没人看也没人维护 |
| **没有理由** | 只描述架构是什么，不说明为什么这样设计 |

### 6.3 推荐工具

| 工具 | 适用场景 |
|:---|:---|
| **Draw.io / Excalidraw** | 快速手绘架构图 |
| **PlantUML** | 代码化 UML 图 |
| **C4 Model** | 比 4+1 更现代的分层视图（Context→Container→Component→Code） |
| **Structurizr** | C4 Model 的专业工具 |
| **Arc42** | 标准化架构文档模板 |

**17.** 解释逻辑视图中的"类图"和开发视图中的"包图"为什么容易混淆？区分它们的关键判断标准是什么？

<details><summary>参考</summary>

**混淆原因**：
- 类图和包图在 UML 中都是结构图，都描述"静态关系"
- 在实际项目中，包往往就是类的容器，容易出现"包图就是类的分组"的误解

**区分标准**：

| 维度 | 逻辑视图(类图) | 开发视图(包图) |
|:---|:---|:---|
| **关注点** | "业务概念是什么" | "代码怎么组织" |
| **元素** | 领域实体、值对象、服务接口 | 模块、包、子系统、jar |
| **关系** | 继承、关联、聚合、依赖（领域语义） | 编译依赖、分层关系（工程语义） |
| **受众** | 业务分析师、领域专家 | 开发人员、DevOps |

**关键判断**：如果图描述的是"订单包含多个订单项"这样的业务关系 → 逻辑视图。如果图描述的是"order-module 编译依赖 payment-api"这样的工程关系 → 开发视图。
</details>

**18.** 架构文档中，"Architecture Decision Record (ADR)"的作用是什么？请写一个 ADR 的示例。

<details><summary>参考</summary>

**ADR 的作用**：
记录"为什么选择了这个方案而不是另一个"——保存架构决策的上下文和理由，避免后来者质疑"当初为什么这么设计"。

**ADR 示例**：

```markdown
# ADR-001: 选择 Redis 作为缓存层

## 状态
已采纳（2026-05-27）

## 背景
系统需要缓存热点数据以提升查询性能，需要选型缓存方案。

## 决策
选择 Redis 而非 Memcached。

## 理由
1. Redis 支持更丰富的数据结构（List、Set、Sorted Set）
2. 支持持久化（RDB + AOF），可在重启后恢复数据
3. 团队有 Redis 运维经验
4. 社区活跃，生态完善

## 后果
- 正面：丰富的功能减少了应用层逻辑
- 负面：单线程模型下需注意慢查询阻塞
- 风险：主从切换时存在数据不一致窗口（已通过 TTL + 业务补偿缓解）
```
</details>

**1. 说服团队的 3 个理由**：

- **新人 onboarding**："一个新同事入职，光看代码弄清楚 30 个服务的关系需要 2 周。有架构文档（至少系统上下文图和容器图），半天就能理解全局。"

**2. 降低文档成本的策略**：

- **够用就好，不求完美**：C4 模型的 Level 1（系统上下文图）+ Level 2（容器图）通常是 80% 场景下需要的全部

- **聚焦关键决策**：不需要描述每个接口的参数细节，但必须记录 ADR（为什么微服务拆分是 10 个而不是 20 个）
