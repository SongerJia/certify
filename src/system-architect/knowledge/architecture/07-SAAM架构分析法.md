## 二、SAAM 概述

| 项目 | 内容 |
|:---|:---|
| **全称** | Software Architecture Analysis Method |
| **中文** | 软件架构分析方法 |
| **提出者** | Kazman, Abowd, Bass, Clements (1994) |
| **定位** | 最早的通用架构评估方法，ATAM 的前身 |
| **核心思想** | 用**场景**驱动评估，检查架构是否满足需求 |

### SAAM 的评估哲学

> 不追求"最优"，而是**验证架构是否满足关键场景需求**。

```
输入场景 → 在架构上走查场景 → 评估满足程度 → 发现问题
```

## 三、SAAM 六步法

```
Step 1: 开发场景 (Scenario Development)
        ↓
Step 2: 描述架构 (Architecture Description)
        ↓
Step 3: 场景分类与优先级 (Classification & Prioritization)
        ↓
Step 4: 场景评估 (Scenario Evaluation) ← 核心
        ↓
Step 5: 场景交互分析 (Scenario Interaction) ← SAAM 独创
        ↓
Step 6: 整体评估报告 (Overall Evaluation)
```

### Step 1：开发场景

> 收集利益相关者关心的所有使用场景

场景来源：
- 用例文档
- 质量属性需求
- 变更历史
- 利益相关者头脑风暴

### Step 2：描述架构

> 用统一的方式描述候选架构方案

描述内容：
- 组件及其职责
- 组件之间的连接关系
- 数据流和控制流
- 部署视图

### Step 3：场景分类与优先级

| 分类 | 说明 | 例子 |
|:---|:---|:---|
| **直接场景** | 架构直接支持 | "用户登录" |
| **间接场景** | 需要部分修改 | "增加第三方登录" |

对每个场景设定优先级（高/中/低）。

### Step 4：场景评估（核心步骤）

> 对每个场景，在架构上"走查"，判断：

| 评估内容 | 说明 |
|:---|:---|
| **是否支持** | 当前架构是否支持该场景 |
| **实现方式** | 需要哪些组件参与 |
| **修改影响** | 多大范围的代码需要改动 |
| **复杂度** | 实现该场景的难度 |

### Step 5：场景交互分析（SAAM 独创）

> 这是 SAAM 区别于 ATAM 的最关键特点

**核心问题**：不同场景修改的是否是**同一组组件**？

```
场景A（增加支付方式）→ 需要改 订单模块 + 支付模块
场景B（增加物流方式）→ 需要改 订单模块 + 物流模块
                         ↑
                     共享组件 = 订单模块
                     ⚠ 场景交互！频繁修改同一组件是风险点
```

**场景交互矩阵**：

|  | 场景A | 场景B | 场景C |
|:---|:---:|:---:|:---:|
| **订单模块** | ✅ 影响 | ✅ 影响 | - |
| **支付模块** | ✅ 影响 | - | - |
| **物流模块** | - | ✅ 影响 | - |
| **用户模块** | - | - | ✅ 影响 |

> **解读**：订单模块被两个场景"共享"，是关注点分离不足的信号。

### Step 6：整体评估

输出：
- 各场景满足程度
- 场景交互分析（结构划分问题）
- 改进建议

## 四、SAAM vs ATAM 详细对比

| 维度 | SAAM | ATAM |
|:---|:---|:---|
| **诞生时间** | 1994 （更早） | 2000 （SAAM 的演进） |
| **步骤数** | 6 步 | 9 步 |
| **核心输出** | 场景满足度 + 场景交互矩阵 | 风险/权衡/敏感点 + 效用树 |
| **质量属性处理** | 场景驱动（隐含） | 效用树（显式建模） |
| **权衡分析** | 弱 | 强（核心特色） |
| **场景交互分析** | ✅ **独有** | ❌ |
| **利益相关者参与** | 较少 | 多轮参与 |
| **复杂度** | 中等 | 高 |
| **适用场景** | 快速评估、候选方案对比 | 重大架构决策、多质量属性权衡 |

### 选择建议

| 情况 | 推荐方法 |
|:---|:---|
| 快速评估单个架构方案 | SAAM |
| 多个候选方案对比 | SAAM |
| 重大架构决策，涉及多方利益 | ATAM |
| 需要深度分析质量属性权衡 | ATAM |
| 发现组件划分/关注点分离问题 | SAAM（场景交互分析） |

> **考试要点**：对比题常考"SAAM 有场景交互分析，ATAM 没有"和"ATAM 显式处理质量属性权衡，SAAM 没有"。

## 五、SAAM 实战演练

### 场景：在线商城架构评估

**架构方案**：分层架构（表示层 → 业务层 → 数据层）

**待评估场景**：

| 场景 | 描述 | 优先级 |
|:---|:---|:---:|
| S1 | 用户下单购买商品 | 高 |
| S2 | 增加团购功能（新业务模式） | 高 |
| S3 | 接入微信支付（新支付渠道） | 中 |
| S4 | 数据迁移到新数据库 | 中 |

### Step 4 场景评估

| 场景 | 支持？ | 涉及组件 | 修改范围 |
|:---|:---:|:---|:---|
| S1 | ✅ | 表示层 + 业务层 + 数据层 | 无（已实现） |
| S2 | ⚠️ 需修改 | 业务层 + 数据层 | 中等 |
| S3 | ⚠️ 需修改 | 业务层（支付模块） | 小 |
| S4 | ⚠️ 需修改 | 数据层 | 中等 |

### Step 5 场景交互分析

| 模块 | S1 | S2 | S3 | S4 | 交互次数 |
|:---|:---:|:---:|:---:|:---:|:---:|
| 表示层 | ✅ | - | - | - | 1 |
| 业务层 | ✅ | ✅ | ✅ | - | **3** ⚠️ |
| 数据层 | ✅ | ✅ | - | ✅ | **3** ⚠️ |

> **结论**：业务层和数据层被多个场景频繁修改，暗示关注点分离可以进一步优化。建议考虑将业务层按领域拆分（如订单域、支付域、商品域）。

**17.** 如果你要为"业务层和数据层被多个场景频繁修改"的系统提出改进建议，你会推荐什么方向？请从 SAAM 的角度解释。

<details><summary>参考</summary>

SAAM 视角的改进建议：

1. **关注点分离**：将业务层按领域拆分为订单域、支付域、商品域（DDD 界限上下文）
2. **引入抽象层**：在业务层和数据层之间增加仓储（Repository）或 DAO 接口，隔离数据层变更影响
3. **遵循开闭原则**：对扩展开放、对修改关闭——新增场景应通过扩展而非修改现有模块来实现

SAAM 核心理念：**场景驱动的架构演进**——不是预先设计完美的架构，而是通过分析"哪些变更最频繁"来指导架构优化方向。
</details>

**18.** 对比 SAAM 和 ATAM 在实践中的选型建议：什么情况下选 SAAM，什么情况下选 ATAM？

| 选择 SAAM | 选择 ATAM |
|:---|:---|
| 评估时间 < 1 天 | 评估时间 ≥ 2 天 |
| 主要关注可修改性 | 需要覆盖多个质量属性 |
| 需要快速发现架构修改热点 | 需要系统性风险管理 |
| 团队对架构评估方法不熟悉（SAAM 更简单） | 有经验的评估团队 |
| 遗留系统接手/重构前的快速诊断 | 新系统的正式架构评审 |
| CEO 明天要方案对比 | 架构委员会季度评审 |
</details>

**20.** 某系统目前架构如下：表示层（Web/iOS）、业务逻辑层（单体）、数据访问层、MySQL。现需评估是否引入"缓存层"和"消息队列"。请构建 SAAM 场景交互矩阵来分析这个变更的影响范围。

<details><summary>参考</summary>

**评估场景**：
- S1：新增 Redis 缓存加速热门查询
- S2：引入 MQ 实现订单异步处理
- S3：缓存与数据库数据不一致的处理
- S4：MQ 消息丢失的补偿机制
- S5：缓存穿透/雪崩的防护

**场景交互矩阵**：

| | 表示层 | 业务逻辑层 | 数据访问层 | 缓存层(新) | MQ层(新) | MySQL |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|
| S1 新增Redis | ✓ | ✓ | ✓ | ● | | |
| S2 引入MQ | | ✓ | | | ● | |
| S3 缓存不一致 | | ✓ | ✓ | ● | | |
| S4 MQ消息丢失 | | ✓ | | | ● | |
| S5 缓存防护 | | ✓ | | ● | | |

● = 新增模块 | ✓ = 需要修改 | 空格 = 无需修改

**分析结论**：
- **热点模块**：业务逻辑层被 5 个场景全部标记 → 职责过多，建议拆分缓存策略和 MQ 处理为独立服务
- **新增模块**：缓存层和 MQ 层是新引入的架构元素
- **数据访问层**：S1 和 S3 涉及 → 建议在数据访问层增加一层抽象（缓存中间件）
- **表示层**：仅在 S1 涉及（API 响应格式可能变化）→ 影响可控
</details>
