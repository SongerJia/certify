## 二、质量属性场景的结构

每个质量属性场景由 6 个部分组成：

```
刺激源 (Source) → 刺激 (Stimulus) → 制品 (Artifact) → 响应 (Response) → 响应度量 (Measure)
                    ↑
               环境 (Environment)
```

| 组成部分 | 含义 | 例子 |
|:---|:---|:---|
| **刺激源** | 谁/什么产生了刺激 | 用户、攻击者、硬件故障 |
| **刺激** | 发生了什么 | 提交订单、DDOS 攻击、磁盘损坏 |
| **环境** | 什么条件下发生 | 正常运行、高峰期、降级模式 |
| **制品** | 系统的哪个部分受影响 | 整个系统、某个服务、数据库 |
| **响应** | 系统如何反应 | 处理请求、切换到备库、降级 |
| **响应度量** | 如何衡量响应质量 | 3 秒内返回、RPO &lt; 1 分钟 |

> **记忆口诀**：**源刺环制响度** — 谁在什么环境下对什么产生了什么刺激，导致什么反应，怎么衡量？

## 三、六大质量属性

### 3.1 可用性 (Availability)

**定义**：系统在需要时能正常运行并提供服务的能力。

| 维度 | 内容 |
|:---|:---|
| **刺激源** | 内部故障（Bug）、外部故障（断电、硬件损坏） |
| **刺激** | 故障发生 |
| **环境** | 正常运行或降级模式 |
| **制品** | 计算资源、网络、存储 |
| **响应** | 检测故障 → 恢复/切换到冗余 |
| **响应度量** | MTBF（平均故障间隔）、MTTR（平均修复时间）、可用率百分比 |

| 可用率 | 年宕机时间 | 常见方案 |
|:---|:---|:---|
| 99% ("两个9") | 3.65 天 | 单机 |
| 99.9% ("三个9") | 8.76 小时 | 主备 |
| 99.99% ("四个9") | 52.6 分钟 | 多活 |
| 99.999% ("五个9") | 5.26 分钟 | 异地多活 |

**可用性战术**：

| 大类 | 战术 | 说明 |
|:---|:---|:---|
| **故障检测** | Ping/Echo | 心跳检测 |
| | 心跳 (Heartbeat) | 定时发送状态 |
| | 异常检测 | 监控指标超阈值 |
| **故障恢复** | 主动冗余 | 多副本同时运行 |
| | 被动冗余 | 主备切换 |
| | 备件 (Spare) | 热备/冷备 |
| **故障预防** | 事务 | 原子性保证 |
| | 进程监视器 | 自动重启 |

---

### 3.2 性能 (Performance)

**定义**：系统在给定时间内处理请求的能力。

| 维度 | 内容 |
|:---|:---|
| **刺激源** | 用户请求、定时任务、外部系统调用 |
| **刺激** | 大量并发请求 |
| **环境** | 正常运行 |
| **制品** | 系统 |
| **响应** | 处理请求 |
| **响应度量** | 响应时间、吞吐量（TPS/QPS）、延迟 |

| 指标 | 含义 |
|:---|:---|
| **响应时间** | 从请求发起到收到响应的时间 |
| **吞吐量** | 单位时间内处理的请求数 |
| **延迟** | 数据在网络/系统中传输的耗时 |
| **并发数** | 同时处理的请求数 |

**性能战术**：

| 大类 | 战术 | 说明 |
|:---|:---|:---|
| **资源需求** | 提高计算效率 | 算法优化 |
| | 减少计算开销 | 减少不必要的处理 |
| **资源管理** | 引入并发 | 多线程/异步 |
| | 增加资源 | 水平/垂直扩展 |
| | 维护数据副本 | 缓存 |
| **资源仲裁** | 调度策略 | FIFO/优先级/轮询 |

---

### 3.3 安全性 (Security)

**定义**：系统抵抗未授权访问、保证数据机密性/完整性/可用性的能力。

| 维度 | 内容 |
|:---|:---|
| **刺激源** | 攻击者（外部/内部） |
| **刺激** | 未授权访问、篡改、拒绝服务 |
| **环境** | 在线运行，可能受攻击 |
| **制品** | 数据、服务、通信通道 |
| **响应** | 认证、授权、加密、审计 |
| **响应度量** | 攻击难度、恢复时间、损失范围 |

**CIA 三要素**：

```
机密性 (Confidentiality)  — 数据不被未授权者访问
完整性 (Integrity)        — 数据不被未授权篡改
可用性 (Availability)     — 授权用户能正常访问
```

**安全性战术**：

| 大类 | 战术 | 说明 |
|:---|:---|:---|
| **抵御攻击** | 认证 | 验证身份 |
| | 授权 | 控制访问权限 |
| | 加密 | 数据机密性 |
| **检测攻击** | 入侵检测 (IDS) | 监控异常行为 |
| | 审计追踪 | 记录操作日志 |
| **从攻击中恢复** | 恢复状态 | 回滚到安全状态 |
| | 识别攻击者 | 溯源追踪 |

---

### 3.4 可修改性 (Modifiability)

**定义**：系统能以成本有效的方式响应变更的能力。

| 维度 | 内容 |
|:---|:---|
| **刺激源** | 开发者、业务变更 |
| **刺激** | 需求变更、技术升级 |
| **环境** | 开发/运维/运行时 |
| **制品** | 代码、配置、架构 |
| **响应** | 修改并测试变更 |
| **响应度量** | 修改范围、修改成本、测试工作量 |

**可修改性战术**：

| 大类 | 战术 | 说明 |
|:---|:---|:---|
| **局部化修改** | 高内聚 | 相关功能集中 |
| | 预见的变更点 | 策略模式、配置化 |
| **防止连锁反应** | 信息隐藏 | 封装实现 |
| | 使用中介 | 接口隔离 |
| **推迟绑定时间** | 运行时绑定 | 插件、依赖注入 |
| | 配置文件 | 外部化配置 |

---

### 3.5 可测试性 (Testability)

**定义**：通过测试揭示系统缺陷的难易程度。

| 战术 | 说明 |
|:---|:---|
| **输入/输出分离** | 明确接口边界 |
| **记录/回放** | 捕获状态用于回归测试 |
| **接口与实现分离** | 依赖注入，Mock 测试 |
| **专门测试路径** | 提供内部状态访问 |

---

### 3.6 易用性 (Usability)

**定义**：用户学习、操作系统的容易程度。

| 战术 | 说明 |
|:---|:---|
| **分离用户界面** | UI 与逻辑解耦（MVC/MVP/MVVM） |
| **支持用户主动** | 撤销 (Undo)、取消 (Cancel) |
| **支持系统主动** | 进度条、预估时间、上下文帮助 |

## 四、质量属性场景实战

### 场景识别训练

阅读以下需求，判断对应哪个质量属性：

| # | 需求描述 | 质量属性 |
|:---:|:---|:---:|
| 1 | "系统需要支持 10,000 用户同时在线" | ？ |
| 2 | "系统 7×24 小时运行，允许年度维护窗口 4 小时" | ？ |
| 3 | "添加快递方式时，不应修改下单模块代码" | ？ |
| 4 | "用户密码以不可逆算法存储" | ？ |
| 5 | "BUG 修复后，应能快速验证修复是否引入新问题" | ？ |
| 6 | "新用户 5 分钟内完成首次下单" | ？ |

<details><summary>答案</summary>1. 性能（并发用户数 = 吞吐量指标）<br>2. 可用性（年度可用率 ≈ 99.95%，约三个9到四个9）<br>3. 可修改性（变更局部化，不产生连锁反应）<br>4. 安全性（机密性，密码不可逆存储 = 哈希加密）<br>5. 可测试性（快速验证修复 = 测试效率）<br>6. 易用性（新用户体验 = 操作效率）</details>

## 六、章节练习（25 题）

### 单选题（15 题）

**1.** 系统在遭遇 DDoS 攻击后，能在 5 分钟内自动切换到备用数据中心。这描述的是哪个质量属性的响应度量？
- A. 性能
- B. 可用性
- C. 安全性
- D. 可修改性

<details><summary>答案</summary>B — 故障恢复时间是可用性的核心度量</details>

**2.** "增加一个新的支付渠道时，只需新增一个支付插件，不需要修改订单模块代码"——体现了哪个质量属性？
- A. 性能
- B. 可用性
- C. 可修改性
- D. 安全性

<details><summary>答案</summary>C — 变更影响范围控制是可修改性的核心</details>

**3.** 以下哪个不是性能战术？
- A. 引入缓存
- B. 异步处理
- C. 主动冗余
- D. 负载均衡

<details><summary>答案</summary>C — 主动冗余是可用性战术，用于故障恢复而非性能优化</details>

**4.** 质量属性场景的六部分结构中，"制品"（Artifact）指的是？
- A. 触发质量属性响应的来源
- B. 需要保障质量的系统组件
- C. 系统的外部环境
- D. 度量响应的方法

<details><summary>答案</summary>B — 制品是被质量属性保护的系统部分（如"支付系统"、"数据库"）</details>

**5.** "用户发起支付请求时，系统在 3 秒内完成处理并返回结果"——六部分中的"响应度量"是？
- A. 用户发起支付请求
- B. 支付系统
- C. 3 秒内
- D. 完成处理并返回结果

<details><summary>答案</summary>C — 响应度量是对响应效果的可量化衡量</details>

**6.** 以下哪种战术属于"可用性"的故障检测类战术？
- A. 主动冗余
- B. Ping/Echo 心跳检测
- C. 负载均衡
- D. 限流降级

<details><summary>答案</summary>B — 心跳检测是可用性三大战术分类中的"故障检测"类</details>

**7.** 安全性战术中的"抵御攻击"不包括以下哪项？
- A. 身份认证
- B. 数据加密
- C. 入侵检测
- D. 缓存预热

<details><summary>答案</summary>D — 缓存预热是性能战术，与安全无关</details>

**8.** "当系统的用户量从 10 万增长到 100 万，响应时间从 200ms 变为 500ms，仍在 SLA 范围内"。这最能体现哪个可修改性相关质量属性？
- A. 可修改性
- B. 可扩展性（Scalability）
- C. 可用性
- D. 可测试性

<details><summary>答案</summary>B — 可扩展性衡量系统在负载增长后的性能表现，属于性能质量属性的子维度</details>

**9.** 关于"互操作性"（Interoperability），描述正确的是？
- A. 等同于可用性
- B. 系统与其他系统交换和使用信息的能力
- C. 仅适用于 SOA 架构
- D. 是安全性的子属性

<details><summary>答案</summary>B — 互操作性是 ISO 25010 中的独立质量属性，衡量跨系统协作能力</details>

**10.** 可用性战术中的"故障恢复"类不包括？
- A. 状态同步
- B. 回滚（Rollback）
- C. 重试（Retry）
- D. 身份认证

<details><summary>答案</summary>D — 身份认证属于安全性战术的"抵御攻击"类</details>

**11.** 可修改性战术的核心思路是？
- A. 提高系统吞吐量
- B. 降低变更的影响范围（局部化修改）
- C. 增加系统的冗余度
- D. 减少用户的操作步骤

<details><summary>答案</summary>B — 可修改性的本质是"控制变更成本"，通过模块化、封装、抽象来局部化修改影响</details>

**12.** 以下哪种做法属于"可测试性"战术？
- A. 使用微服务架构
- B. 增加监控告警
- C. 暴露内部状态接口供测试访问
- D. 使用 HTTPS

<details><summary>答案</summary>C — 可测试性战术的核心是让系统状态可观测、可控制</details>

**13.** 性能战术中"资源管理"类不包括？
- A. 并发处理（多线程）
- B. 数据缓存
- C. 心跳检测
- D. 连接池

<details><summary>答案</summary>C — 心跳检测是可用性战术</details>

**14.** 可用性和安全性的 CIA 模型中都有"可用性"一词，它们的不同是？
- A. 完全相同
- B. 架构可用性关注系统不挂，CIA 可用性关注合法访问不被阻断
- C. 架构可用性只关注硬件，CIA 只关注软件
- D. 没有本质区别

<details><summary>答案</summary>B — 架构可用性 = 系统 uptime；CIA 可用性 = 授权用户能正常访问数据</details>

**15.** 某系统在"双11大促"期间：每秒 10 万请求 → 响应时间 < 200ms。六部分中"刺激"是什么？
- A. 双11大促期间
- B. 每秒 10 万请求
- C. 响应时间 < 200ms
- D. 用户

<details><summary>答案</summary>B — 刺激是到达系统的事件（每秒 10 万请求），"双11大促期间"是环境</details>

**18.**（多选）关于质量属性场景，正确的说法有？

- A. 一个场景只描述一个质量属性

- C. 两者完全独立，不能组合使用
- D. 资源需求战术包括引入并发

<details><summary>答案</summary>A、B — 两种战术常组合使用（压缩数据 + 连接池并行发送）。引入并发属于资源管理类。</details>

**21.** 某系统需求："当数据库主库故障时，系统自动切换到备库，切换时间不超过 30 秒，且切换过程中未提交的事务自动回滚"。请按质量属性场景六部分拆解。

<details><summary>参考</summary>

| 六部分 | 对应内容 |
|:---|:---|
| **刺激源**（Source） | 数据库主库 |

| **制品**（Artifact） | 数据库系统（主备集群） |
| **响应**（Response） | 自动切换到备库，未提交事务回滚 |
| **响应度量**（Measure） | 30 秒内完成切换 |

</details>

**22.** 解释"可修改性"与"可维护性"的异同，并说明在日常架构评估中为什么常将二者合并讨论。

<details><summary>参考</summary>

| 维度 | 可修改性 | 可维护性 |
|:---|:---|:---|
| **焦点** | 代码变更的难易程度 | 系统运维的难易程度 |
| **典型场景** | 新增功能、修改业务规则 | 修复 Bug、升级依赖、排查问题 |
| **战术** | 模块化、抽象、封装 | 日志、监控、健康检查、文档 |

**常合并讨论的原因**：
1. 二者战术高度重叠（好的模块化同时提升修改性和维护性）
2. 考试中通常将可维护性归入可修改性的范畴
3. 实践中"改代码"和"查问题"经常交替发生，界限模糊

</details>

**25.** 某电商平台在"618 大促"前的架构评审中，团队对以下两个需求有争议：
- 需求 X：为应对流量峰值，提前扩容到 5 倍日常容量
- 需求 Y：为防止超卖，所有库存扣减必须强一致（不允许脏读）

这两个需求之间可能存在什么冲突？作为架构师，你如何分析这个权衡并提出折中方案？

<details><summary>参考</summary>

**冲突分析**：
- 需求 X（性能/可扩展性）→ 倾向分布式、异步处理、多副本
- 需求 Y（一致性/可用性）→ 倾向集中式锁、串行化、单点扣减
- **核心矛盾**：分布式多副本（性能高） vs 串行化强一致（性能低）

**权衡分析**（类 ATAM 思维）：

| 维度 | 全分布式方案 | 全集中方案 | 折中方案 |
|:---|:---|:---|:---|
| 性能 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| 一致性 | ⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 复杂度 | ⭐ | ⭐⭐⭐ | ⭐⭐ |

**折中方案——热点隔离**：
1. **普通商品**：Redis 分布式扣减（最终一致性可接受，99% 商品不会超卖）
2. **秒杀/热门商品**：独立库存服务 + 数据库行级锁（强一致，但只针对少量热点商品）
3. 大促前通过历史数据预判热点商品列表，提前配置

**效用树分析**：
- (H,H) 性能/大促吞吐：普通商品 Redis 扣减，TPS > 目标值
- (H,H) 一致性/超卖风险：热门商品 DB 强一致，超卖概率 = 0
- 权衡点：性能和一致性在"热点商品"上仍然互斥，通过限制热点范围来缓解
</details>
