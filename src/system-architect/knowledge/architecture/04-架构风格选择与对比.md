| **微内核** | ★ | ★★ | ★★★ | ★★ | ★★★ | ★★★ |

### 2.2 非功能需求 → 风格匹配图谱

```
需求："我需要可扩展的插件机制"      →  微内核
需求："系统需要流式处理大量数据"    →  管道-过滤器
需求："各模块需要异步松耦合"        →  事件驱动
需求："需要支持多客户端统一API"     →  分层架构
需求："跨系统集成，异构技术栈"      →  SOA
需求："团队并行开发，独立部署"      →  微服务
需求："UI 逻辑复杂，需数据双向同步" →  MVVM
```

## 三、场景选型决策树

### 决策流程

```
1. 确定核心质量目标（1-2个最重要的）
        │
2. 匹配质量属性矩阵（定位候选风格范围）
        │
3. 考虑约束条件（团队、技术栈、时间、遗留系统）
        │
4. 列举 2-3 个候选方案
        │
5. 用 ATAM/SAAM 评估候选方案
        │
6. 选择最优方案，记录权衡点
```

### 经典场景与推荐

| 场景描述 | 推荐风格 | 备选 | 核心权衡 |
|:---|:---|:---|:---|
| 企业 MIS 系统 (CRUD 为主) | 分层架构 | SOA | 简单性 vs 灵活性 |
| 实时数据流处理 | 管道-过滤器 | 事件驱动 | 吞吐量 vs 延迟 |
| IoT 设备管理平台 | 事件驱动 + 微服务 | 分层 | 松耦合 vs 一致性 |
| IDE/开发平台 | 微内核 | 分层 | 扩展性 vs 性能 |
| 电商系统 | 微服务 | 分层 | 独立部署 vs 简单性 |
| 跨组织系统集成 | SOA | 微服务 | 标准化 vs 灵活性 |
| 内容管理/博客 | 分层架构 | 微内核 | 简单性 vs 扩展性 |
| 社交媒体 Feed 流 | 事件驱动 | 微服务 | 实时性 vs 一致性 |

## 五、考试常见陷阱与辨析

### 5.1 易混淆对

| 易混淆 | 区分方法 |
|:---|:---|
| **分层 vs SOA** | 分层层内紧耦合，SOA 服务间松耦合 |
| **SOA vs 微服务** | SOA 有 ESB 中心，微服务去中心化 |
| **事件驱动 vs 管道-过滤器** | 事件驱动响应事件，管道过滤处理数据流 |
| **微内核 vs 分层** | 微内核核小插件多，分层各层对等 |
| **MVC vs MVP** | MVC 中 View 知 Model，MVP 中 View 不知 Model |

### 5.2 常见错误说法

- ❌ "微服务一定比单体好"
- ❌ "分层的层数越多越好"
- ❌ "事件驱动就是异步处理"
- ❌ "SOA 已经过时，被微服务完全取代"
- ❌ "选择了某风格就不能混用其他风格"

### 5.3 正确观点

- ✅ 架构风格可以混合使用（如微服务内部使用分层）
- ✅ 没有"最好"的风格，只有"最合适"的权衡
- ✅ 分层越严格，维护性越好但性能越差
- ✅ SOA 和微服务各有所长，大型企业仍广泛使用 SOA

- A. 管道-过滤器

**15.** 解释"Conway's Law"（康威定律）对架构选型的影响。

<details><summary>参考</summary>

康威定律：**系统的设计结构会镜像其组织的沟通结构。**

对架构选型的影响：
- 如果团队按功能模块划分（订单组、支付组、商品组），微服务架构天然匹配这种组织结构
- 如果团队是一个紧密协作的小团队，单体架构可能更高效
- **推论**：改变架构之前，可能需要先调整团队结构（逆向康威定律）

举例：想从单体迁到微服务，但团队仍然是"所有人改所有代码"的模式 → 微服务一定会失败。
</details>

**16.** 架构选型中如何平衡"过度设计"和"设计不足"？请给出一个实用判断框架。

<details><summary>参考</summary>

**实用判断框架**：

| 判断维度 | 过度设计信号 | 设计不足信号 |
|:---|:---|:---|
| 需求确定性 | 需求模糊却做精细化架构 | 需求明确却无架构设计 |
| 规模预期 | 面向百万用户设计，实际只有 100 用户 | 已知会快速扩容却没有扩展性设计 |
| 团队能力 | 选了团队不熟悉的技术 | 所有问题都用"再加一个 if"解决 |

**实用原则**：满足当前需求 + 预留 1 个数量级的增长空间 + 清晰的模块边界（方便未来重构）
</details>

**18. 案例分析**：某在线教育平台需求如下——Web/iOS/Android 三端、课程直播（延迟 < 3s）、视频回放（百万并发）、在线考试（数据不丢失）、团队 50 人 5 组。请设计顶层架构并解释选型理由。

**19.** 某公司考虑从单体架构迁移到微服务。请列出 3 个"不要迁"的合理理由，以及 3 个"应该迁"的充分信号。

<details><summary>参考</summary>

**不要迁的理由**：
1. 团队 < 10 人，单体架构开发和运维效率更高
2. 业务还处于验证期，需求变化剧烈，过早拆分会导致频繁的接口变更成本
3. 没有运维微服务的能力（容器化、监控、CI/CD、服务治理）

**应该迁的信号**：
1. 不同模块的发布节奏严重冲突（一个模块改一行需要等另一个模块的 2 周测试）
2. 部分模块有独立的扩展需求（搜索模块需要 10 台机器，其他模块 2 台就够）
3. 团队膨胀到无法在一个代码库中高效协作
</details>

**20.** 设计一个医疗影像处理系统：影像上传 → AI 辅助诊断 → 医生审核 → 生成报告 → 归档。AI 诊断可能需要数分钟，医生审核可能数小时。请设计架构风格组合并画出数据流。

<details><summary>参考</summary>

**推荐组合：事件驱动 + 管道-过滤器 + 仓库**

```
影像上传
  ↓ (事件驱动)
影像预处理（管道-过滤器）
  ├→ 格式标准化 → 质量检测 → 写入影像仓库
  ↓ (事件通知)
AI诊断服务（异步，数分钟）
  → 诊断结果写入"诊断仓库"
  ↓ (事件通知)
医生审核（异步，数小时）
  → 审核结果写入"诊断仓库"
  ↓
报告生成（管道-过滤器：诊断数据→模板渲染→PDF）
  ↓
归档（写入长期存储）
```

| 阶段 | 风格 | 理由 |
|:---|:---|:---|
| 上传→预处理 | 管道-过滤器 | 标准的数据处理管线 |
